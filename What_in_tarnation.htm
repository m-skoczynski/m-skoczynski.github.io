What in tarnation

This documentation covers the RES_PROCESSOR scripts package developed by REDDO Translations.

Who is this for

This documentation is meant for two groups of users: developers who will be maintaining, developing or troubleshooting RES_PROCESSOR, as well as operators who need to make use of the features offered by the scripts package.

If you are unsure which group you should be in, please refer to the "Dramatis personae" section of this documentation here.

Why do I need this

The purpose of this documentation is to explain the structure of the code behind RES_PROCESSOR to facilitate maintenance and further development.

It also explains the procedure the operator needs to follow to prepare input data for the scripts to maintain light and heavy resources on a MemoQ server, as well as the actual steps to be performed when using the software.

There is also a Troubleshooting section, which is updated with all the exceptional workflows that need to be followed depending on the situation in a particular project.

 

A bit of history

In case you did not notice yet, REDDO Translations is a translation company. Translation workflows at REDDO generate two type of "heavy" resources: Translation Memories and Term Bases.

Every translation job REDDO does is handled as an Order, which has it's own Plunet workflows, MemoQ project, as well as a set of heavy resources - this separation facilitates keeping an audit trail, managing current work but also ensuring quality.

Since every bit of translation is handled separately, REDDO is able to ensure that if any errors have been committed in the translation, they are not "propagated" onto other translation work.

Keeping resources separate also allows to make sure proper context is provided - every heavy resources is meta-tagged with the client [i.e. company name, referring to the end client], domain [i.e. general topic area, e.g. medical, legal, technical] and subject [i.e. more specific sub-area: agreement, certificate, manual].
