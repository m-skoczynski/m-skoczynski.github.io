Process overview

This section gives you a general overview of what the process behind PLU_ARCHIVER is.

Business scope

 

 

Pure and simple - the business translates, generating heavy resources and Plunet orders.

There is no need for any special process, this happens in the course of business-as-usual.

 

 

 

PM scope

The PM needs to make a decision on whether a scheduled resource maintenance run should be initiated (e.g. a pre-set amount of time has elapsed since last run)

or

whether operational needs warrant an ad-hoc resource maintenance run (e.g. excessive volume of resources).

At times, the PM needs to provide input to the RCL operator if they are unable to resolve maintenance issues with past projects.

This may include:

  • access to alternative information - e.g. project worksheet

  • corrections in Plunet - e.g. target TM

  • deciding on dedicated client resources to be created (though this should happen at the scoping stage)

 

The PM needs to provide input for the process, which includes:

  • scoping - determine what period, domain and subject should be processed

  • XLS/CSV creation - the PM needs to export the validated Plunet orders report BEFORE* changing order status

    * if there are any issues during status change, the XLS/CSV export needs to be updated manually; otherwise, scope will be lost

  • Plunet order/item status - Plunet orders need to be set to the "archived" status and any issues need to be resolved or orders that cannot be archived need to be removed from the scope

  • RCL setup - an RCL job needs to be set up for the proper operator, including scope description and XLS/CSV exports

  • deadlines - proper deadlines need to be set downstream, accounting for production load

Once the RCL operator reports that a maintenance run is complete, the PM needs to plan further workflow stages:

  • what work (if any) needs to be performed by senior resources and when

  • what work needs to be performed by regular resources and when

The output of this step are proper GLO jobs in the appropriate resource maintenance order.

RCL operator scope

The Operator must review the input provided by the PM to confirm the following:

  • number of orders folders matches the number of lines in the CSV file

  • orders in folder match CVS contents (optional; may be done via the MessCleanUpTools.py deprecated tool)

  • XLS and CVS export files are available and contents match

  • XLS/CVS contains all the necessary data (no empty field in Target TM, Client, Order number)\

If no issues are found, RES_PROCESSOR may continue; if issues are found, either rectify independently or circle back to PM.

Once input is reviewed and confirmed OK and the Plunet server prep activities are successfully completed, a series of iterations of the RES/PROCESSOR and PostTools may be performed.

Subsequent iterations that lead to full processing of all the items included in the scope is called a run.

A run provides the following outputs:

  • project TMs/TBs are deleted from the MemoQ server

  • project TMs have been imported into the proper pooled resource on the MemoQ server

  • project TBs have been pooled into a "dump" file per domain and imported to the MemoQ server

  • Plunet orders in ZIP files in a year/month structure have been sent to the archive location

  • audit trail elements including log files have been archived in a ZIP file with run date and topics, sent to the archive location

Before initiating RES_PROCESSOR, the following activities need to be performed on the Plunet server:

  • confirm that the number of archive order folders in the ARCHIVE folder matches the number of lines in CSV (-1 for header)

  • connect to the ARCHIVE folder via RDP (NOT!: Windows drive mapping) and initiate separate ZIP packing for each folder

  • transfer ZIP folders to the production environment (for best performance, connect a physical external drive to the server)

Each iteration of RES_PROCESSOR may result in special cases that need to be resolved manually (more information is available in RES_PROCESSOR step-by-step manual).

 

In most cases, the PostTools.py tool will be able to resolve simple special cases consisting in missing TM/TB export files (this happens often when a TEX job "freezes" due to having to export too many resources/too heavy resources). There are however other instances that need the Operator's manual intervention, also described in more detail in the RES_PROCESSOR step-by-step manual.

Senior/Regular resource scope

Both senior resources (usually Head of Quality/Head of Reviewers) and regular resources (Translators/Reviewers/PMs) may issue an informal resource maintenance request. Junior resources are not allowed to isssue such requests and must process any ideas through a Regular or Senior resource.

Requests for resource maintenance may either result from current production insights (e.g. errors identified, error propagation identified, new preference introduced) or general workflow considerations (e.g. isolating customer resources, isolating low-quality resources, initiating a maintenance run out of schedule).

Senior resources (Head of Quality/Head of Reviewers) will review resource maintenance requests submitted by Regular resources. If the review indicates that a maintenance run is warranted, the Senior resources concerned forwards the request to the PM in charge of resources.

 

The PM liaises with the requestor to determine timeframe and priority, then feeds the request through the standard process.

RES_PROCESSOR automatically processes TM and TB heavy resources: deletes project resources from the MemoQ servers, imports TMs into global TMs and creates TB dump files that require manual "protein-based" processing.

 

Processing the dump TBs is the main, regular type of resource work that needs to be performed by translators, according to a separate process.

Other resource may also be commissioned, but is outside the scope of the RES_PROCESSOR workflow.

Once a translator properly sorts the contents of the dump TB into what is needed and must be re-used in global resources or client-specifi resources and what may be discarded, the results are then reviewed by a Senior resource. Following validation, these are imported into the global TB resources.
